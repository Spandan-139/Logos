# Code Review: "cool project" notebooks (Logos LM, v0–v21)

## Summary

These 22 notebooks (`cool project (1).ipynb` through `v21`) are one continuously-evolving GPT-style language model project called Logos, not 21 separate scripts. Each version appends one more ablation or scaling experiment on top of the last, matching the experiment reports in your `pdfs/` folder almost 1:1: baseline → weight tying → GELU → RMSNorm → SwiGLU → combined variants → shape ablations (deeper/narrower, wider/shallower) → data cleaning → longer training → scaling (13M→124M params) → quantization (Exp19) → efficiency index (Exp20). `v21` breaks from that lineage — it's a leaner standalone notebook that loads the final Exp18 checkpoint and benchmarks it against public Hugging Face models (GPT-2, Pythia, SmolLM2).

Overall quality is good for research/ablation code: correct causal attention (via `F.scaled_dot_product_attention` with `is_causal=True`), correct AMP + GradScaler usage, sensible weight init, and no security red flags anywhere (no `eval`/`exec`, no shell injection, no hardcoded secrets — `subprocess` is called with an argument list, never `shell=True`). The main issues are consistency/maintainability problems that come from copy-pasting a notebook 21 times rather than refactoring into reusable modules.

## Critical Issues

| # | File | Location | Issue | Severity |
|---|------|----------|-------|----------|
| 1 | v20.py | `torch.load(BEST_MODEL_PATH, ...)` (Exp18 final benchmark), `EXP18_BEST_MODEL_PATH` load (Exp19) | `torch.load` is called without `weights_only=True`. Since PyTorch 2.4+, the default is moving toward safer loading, and unpickling is arbitrary-code-execution territory if a checkpoint ever comes from anywhere other than your own run. Low real risk here since you control every checkpoint, but it's the one habit worth fixing before you ever load a file you didn't produce yourself. | 🟡 Medium |
| 2 | v20.py, lines ~904–958 vs ~411–434 | Two parallel implementations of loss/generation benchmarking exist (`estimate_loss()` + `benchmark_generation_latency()` from the training section, and `estimate_loss_for_model()` + `benchmark_generation_for_model()` added for Exp19). They're ~80% identical. `benchmark_generation_for_model` uses `enc.encode(prompt)` while every other place in the same notebook uses `enc.encode_ordinary(prompt)`. `tiktoken`'s `encode()` raises on strings that look like special tokens; `encode_ordinary()` doesn't. If a future prompt ever contains something like `<|endoftext|>`, one benchmark path throws and the other doesn't — same notebook, two behaviors. | 🟡 Medium |
| 3 | v20.py, Exp19 quantization section | `torch.ao.quantization.quantize_dynamic(model_cpu_fp32, {nn.Linear}, dtype=torch.qint8)` is applied to a model whose `lm_head.weight` is tied to `token_embedding.weight`. Quantizing only `nn.Linear` breaks the tie silently — `lm_head` gets its own INT8 copy, `token_embedding` stays untouched, and `weight_tying_active` (checked via `data_ptr()` elsewhere) would report `False` if you checked it on this model. Not a crash, but the "weight tying" story you're measuring efficiency gains against quietly stops being true for the INT8 variant. Worth a one-line comment so future-you doesn't get confused by the checkpoint size math. | 🟡 Medium |

| 1 | all versions | `get_batch()` / `get_batch_for_device()` | These two functions do the same random-window sampling with slightly different signatures (one hardcodes globals, the other takes `target_device`/`block_size` params). Consolidating into one parameterized function would remove the duplication and the risk of them silently drifting apart, which is already starting to happen. | Maintainability |
| 2 | v0–v20 | Cell header comments (`# CELL 3 —`, `# CELL 7 —`) | These numbered comments no longer match the actual cell index after edits (e.g. a comment says "CELL 3" but it's actually notebook cell 2). Harmless, but confusing when debugging by cell number. Consider dropping the manual numbering and just describing what the cell does. | Maintainability |
| 3 | v20.py, training loop | `class PowerMonitor` | `stop()` calls `self.thread.join(timeout=1)`, but the sampling loop sleeps up to `interval` (2.0s) between checks of `self.running`. The join can return before the thread actually exits. Harmless since the thread is `daemon=True`, but if you ever add cleanup logic after `stop()` that assumes the thread is dead, this will bite you. Lower `interval` or `join()` without a timeout that's shorter than the sleep. | Correctness |
| 4 | v20.py, Exp18 final benchmark | `checkpoint = torch.load(BEST_MODEL_PATH, ...)` | No existence check before loading — if training is interrupted before the first `EVAL_INTERVAL` checkpoint save, this throws a raw `FileNotFoundError` with no context. Your own `v21.py` already fixed this pattern (`if not LOGOS_CKPT_PATH.exists(): raise FileNotFoundError("... | 5 | v20.py, data loading | Train/val split (`text[:n]` / `text[n:]`) | The validation set is the trailing 10% of concatenated documents from a streamed, unshuffled dataset pull. If document topics drift over the stream (common with web-scraped corpora), your val loss is measuring generalization to "later" documents, not a random held-out sample. Fine for a toy/ablation project, but worth a one-line caveat in the report if you ever compare absolute perplexity numbers across runs pulled with different `OWT_SAMPLES`. | Correctness |
| 6 | v20.py, training loop | AMP dtype | You already gate `torch.backends.cuda.matmul.allow_tf32 = True`, which implies an Ampere-or-newer GPU. On that hardware, `torch.bfloat16` autocast avoids needing `GradScaler` entirely (no overflow to scale for) and is usually a drop-in swap from `float16`. Not required — your scaler usage is correct — but it would simplify the training cell. | Performance |
| 7 | all versions | Model definition cells | `model = ModelClass().to(device)` and later `model_fp32 = model.__class__().to(device)` reuse the global `model` variable across distant cells (Exp18 training cell defines it, Exp19 quantization cell 15 cells later depends on it still being in scope). This works in a single top-to-bottom run but will silently break or reuse the wrong model if any cell is re-run out of order — a common notebook footgun. Consider passing the model / checkpoint path explicitly instead of relying on notebook global state. | Maintainability |

## What Looks Good

- Causal attention is implemented with `F.scaled_dot_product_attention(..., is_causal=True)` throughout — no manual masking bugs, no accidentally-non-causal leakage.
- Weight tying is done correctly: `lm_head` switches to `bias=False` in the same version that introduces tying (v1), which is exactly what's required for a clean tie.
- `RMSNorm` (v3+) is a correct, standard implementation (`rsqrt(mean(x^2) + eps) * weight`).
- Top-k / top-p (nucleus) sampling in `generate()` is implemented correctly, including the "keep at least one token" shift (`sorted_indices_to_remove[..., 1:] = ...[..., :-1]`) that people often get backwards.
- `val_ppl = math.exp(val_loss) if val_loss < 20 else float("inf")` guards against `OverflowError` from a diverged loss — small defensive touch that a lot of from-scratch training scripts skip.
- Zero security findings: no `eval`/`exec`, no `shell=True`, no hardcoded credentials, `subprocess.check_output` is called with an argument list rather than a shell string.
- `v21.py`'s comparison methodology explicitly favors bits-per-byte over raw perplexity when comparing models with different tokenizers, and says so in a comment — that's the right call and shows you understand why cross-tokenizer perplexity comparisons are misleading.
- The version-to-version progression is genuinely well organized as an ablation study: single-variable changes (GELU alone, RMSNorm alone, SwiGLU alone) before combining winners, then scaling up, then compressing — that's a sound experimental design, not just "throw things at the wall."

## Verdict

**Approve**, with the medium-severity items (checkpoint loading safety, the `encode` vs `encode_ordinary` inconsistency, and the quantization/weight-tying interaction) worth a quick pass before you'd call this "production," but none of them affect the validity of your results. For a from-scratch GPT reimplementation with 21 iterative experiments, the model code itself is clean and the ablation methodology is the strongest part of the project.
